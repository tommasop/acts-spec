# ACTS Core v1.1.0 — Multi-Story, WAL, Maintenance, Proactive Triggers

**Date:** 2026-05-15
**Scope:** Full implementation plan for ACTS v1.1.0
**Status:** Draft

---

## Problem Statement

ACTS v1.0.0 works well for single-story workflows but has these bottlenecks:

1. **No multi-story concurrency** — Single `.story/` directory, no worktree isolation, no cross-story ownership enforcement
2. **Defensive-only triggers** — Triggers only RAISE ABORT; they never create useful state (unblock events, conflict detection, review queues)
3. **No WAL mode** — Concurrent DB access causes SQLITE_BUSY; no `BEGIN IMMEDIATE` wrapping
4. **No maintenance path** — Every task must belong to a story; random bug fixes need ceremony
5. **No changelog** — Story progress is in the DB but not surfaced as human-readable output
6. **Hardcoded version** — Binary version is compile-time constant, not from git tags
7. **JSON escaping bug** — Manual JSON construction in `db.zig` doesn't escape special characters
8. **No story-level gates** — Gates exist only at task level; story merge has no gate protocol
9. **No agent presence** — No way to see which agents are active on which tasks
10. **No gate SLA tracking** — No visibility into review bottlenecks

---

## Design Principles

- **Enforcement-first:** New triggers must be DB-level, un-bypassable (ACTS moat)
- **WAL-native:** All writes use `BEGIN IMMEDIATE`; readers never blocked
- **Backward compatible:** Existing v1.0.0 projects migrate without data loss
- **Progressive enhancement:** Each phase ships independently; no big-bang release

**Transaction wrapping** — add to `Database`:
```zig
pub fn beginImmediate(self: *Database) !void {
    var err_msg: ?[*:0]u8 = null;
    const rc = c.sqlite3_exec(self.db, "BEGIN IMMEDIATE", null, null, &err_msg);
    if (rc != c.SQLITE_OK) {
        if (err_msg) |msg| { c.sqlite3_free(msg); }
        return error.WriteLockFailed;
    }
}

pub fn commit(self: *Database) !void {
    const rc = c.sqlite3_exec(self.db, "COMMIT", null, null, null);
    if (rc != c.SQLITE_OK) return error.CommitFailed;
}

pub fn rollback(self: *Database) void {
    _ = c.sqlite3_exec(self.db, "ROLLBACK", null, null, null);
}
```

Wrap all write operations (`writeState`, `updateTask`, `addGate`, etc.) with `beginImmediate`/`commit`/`rollback`.

### 0.2 JSON Escaping Fix

**File:** `acts-core/src/db.zig`

**Problem:** `readState()` (lines 123-205) and all other JSON-emitting functions use raw `writer.print("...{s}...", .{value})` without escaping. If a task title contains `"` or `\`, the JSON breaks.

**Solution:** Add a helper:
```zig
fn escapeJsonString(allocator: std.mem.Allocator, input: []const u8) ![]u8 {
    var out = std.ArrayList(u8).init(allocator);
    errdefer out.deinit();
    const w = out.writer();
    for (input) |c| {
        switch (c) {
            '"' => try w.writeAll("\\\""),
            '\\' => try w.writeAll("\\\\"),
            '\n' => try w.writeAll("\\n"),
            '\r' => try w.writeAll("\\r"),
            '\t' => try w.writeAll("\\t"),
            else => try w.writeByte(c),
        }
    }
    return out.toOwnedSlice();
}
```

Apply to all string fields in `readState`, `getTask`, `listGates`, `listDecisions`, `getOwnershipMap`, `checkScope`, `listOperations`, `getOperation`.

### 2.2 Cross-Story Triggers

**File:** `acts-core/src/schema.sql`

```sql
-- Prevent two stories from claiming the same file via DONE tasks
CREATE TRIGGER enforce_cross_story_ownership
BEFORE INSERT ON task_files
BEGIN
  SELECT CASE WHEN EXISTS(
    SELECT 1 FROM task_files tf
    JOIN tasks t ON t.id = tf.task_id
    WHERE tf.file_path = NEW.file_path
      AND t.story_id != (SELECT story_id FROM tasks WHERE id = NEW.task_id)
      AND t.status = 'DONE'
  ) THEN RAISE(ABORT,
    'File already owned by a DONE task in another story. ' ++
    'Use acts override request to modify.')
  END;
END;

-- Detect active file conflicts (two stories working on same file)
CREATE TRIGGER detect_file_conflict
AFTER INSERT ON task_files
BEGIN
  INSERT INTO file_conflicts (file_path, story_a, story_b)
  SELECT NEW.file_path,
         (SELECT story_id FROM tasks WHERE id = NEW.task_id),
         t.story_id
  FROM task_files tf
  JOIN tasks t ON t.id = tf.task_id
  WHERE tf.file_path = NEW.file_path
    AND t.story_id != (SELECT story_id FROM tasks WHERE id = NEW.task_id)
    AND t.status IN ('IN_PROGRESS', 'TODO')
    AND NOT EXISTS (
      SELECT 1 FROM file_conflicts fc
      WHERE fc.file_path = NEW.file_path AND fc.resolved = 0
    );
END;

-- Story merge gate: all tasks DONE + all reviews approved
CREATE TRIGGER enforce_story_merge
BEFORE UPDATE OF status ON stories
WHEN NEW.status = 'MERGED' AND OLD.status != 'MERGED'
BEGIN
  SELECT CASE WHEN EXISTS(
    SELECT 1 FROM tasks WHERE story_id = NEW.id AND status != 'DONE'
  ) THEN RAISE(ABORT, 'Cannot merge story: open tasks remain')
  END;
  SELECT CASE WHEN EXISTS(
    SELECT 1 FROM tasks t
    WHERE t.story_id = NEW.id AND t.status = 'DONE'
    AND NOT EXISTS (
      SELECT 1 FROM gate_checkpoints gc
      WHERE gc.task_id = t.id AND gc.gate_type = 'task-review'
      AND gc.status = 'approved'
    )
  ) THEN RAISE(ABORT, 'Cannot merge story: tasks without approved review')
  END;
END;
```

**Zig implementation outline:**

```zig
fn handleStory(allocator: std.mem.Allocator, args: []const []const u8) !void {
    if (args.len < 2) { printUsage(); std.process.exit(1); }
    const subcommand = args[0];
    const story_id = args[1];

    if (std.mem.eql(u8, subcommand, "create")) {
        try handleStoryCreate(allocator, args);
    } else if (std.mem.eql(u8, subcommand, "list")) {
        try handleStoryList(allocator, args);
    } else if (std.mem.eql(u8, subcommand, "switch")) {
        try handleStorySwitch(allocator, args);
    } else if (std.mem.eql(u8, subcommand, "archive")) {
        try handleStoryArchive(allocator, args);
    } else if (std.mem.eql(u8, subcommand, "merge")) {
        try handleStoryMerge(allocator, args);
    } else if (std.mem.eql(u8, subcommand, "graph")) {
        try handleStoryGraph(allocator, args);
    }
}
```

**Worktree creation logic** (`handleStoryCreate`):
1. Parse `--title`, `--from`, `--parent` args
2. Create git worktree: `git worktree add .acts/worktrees/<id> -b story/<id> <from>`
3. Create `.story/` scaffold inside worktree (plan.md, spec.md, sessions/, tasks/)
4. Insert story into DB with `worktree_path`, `branch`, `parent_story`
5. Create `.acts/current` symlink → worktree path
6. Output: "Story <id> created at .acts/worktrees/<id>"

**Story switch** (`handleStorySwitch`):
1. Verify story exists and is not archived
2. Update `.acts/current` symlink
3. Update `active_story` table (delete old, insert new — in transaction)
4. Output: "Switched to story <id>"

**Story archive** (`handleStoryArchive`):
1. Verify all tasks are DONE
2. Set `status = 'ARCHIVED'`, `archived_at = now()`
3. Remove worktree: `git worktree remove .acts/worktrees/<id>`
4. Remove git branch: `git branch -D story/<id>`
5. Delete from `active_story` if active

**Story merge** (`handleStoryMerge`):
1. Verify story merge gate is approved (or create it)
2. Trigger fires `enforce_story_merge` (all tasks DONE, all reviews approved)
3. Compute semver from task labels or `--semver` flag
4. Merge branch into target: `git checkout <target> && git merge story/<id>`
5. Update story: `status = 'MERGED'`, `semver`, `released_at`
6. Generate changelog entry (see Phase 4)
7. Create git tag: `git tag story/<id>-v<semver>`
8. Archive story

### 2.4 Story Resolution Order

All existing commands gain implicit `--story` resolved from:
1. `--story` flag (explicit override)
2. `$ACTS_STORY` env var
3. `.acts/current` symlink → extract story ID from path
4. CWD inside a worktree → extract from path
5. If single story in DB → auto-select
6. Error: "No active story. Run: acts story list"

**Implementation:** Add `resolveStoryId()` helper in `main.zig`:
```zig
fn resolveStoryId(allocator: std.mem.Allocator, explicit: ?[]const u8) ![]const u8 {
    if (explicit) |id| return try allocator.dupe(u8, id);
    if (std.process.getEnvVarOwned(allocator, "ACTS_STORY")) |env| return env;
    // Read .acts/current symlink
    const link = std.fs.cwd().readLink(".acts/current") catch |err| {
        if (err == error.FileNotFound or err == error.SymLinkInvalid) {
            // Check if single story in DB
            return try getSingleStoryId(allocator);
        }
        return err;
    };
    // Extract story ID from path: .acts/worktrees/PROJ-42 → PROJ-42
    const basename = std.fs.path.basename(link);
    return try allocator.dupe(u8, basename);
}
```

## Phase 3: Proactive Triggers & Agent Presence

**Goal:** Triggers that create useful state, not just block invalid state.

  unblocked_by TEXT NOT NULL REFERENCES tasks(id),
  created_at TEXT DEFAULT (datetime('now')),

-- Review queue
CREATE TABLE IF NOT EXISTS review_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id TEXT NOT NULL REFERENCES tasks(id),
  requested_by TEXT,

  assigned_to TEXT,
  status TEXT DEFAULT 'pending'

  agent_id TEXT PRIMARY KEY,
  story_id TEXT REFERENCES stories(id),
  task_id TEXT REFERENCES tasks(id),

  gate_id INTEGER PRIMARY KEY REFERENCES gate_checkpoints(id),
  deadline TEXT NOT NULL,

### 3.2 Proactive Triggers

```sql
-- Auto-fire when a dependency completes
CREATE TRIGGER notify_dependency_unblock
AFTER UPDATE OF status ON tasks
WHEN NEW.status = 'DONE' AND OLD.status != 'DONE'
BEGIN
  INSERT INTO unblock_events (unblocked_task_id, unblocked_by)
  SELECT td.task_id, NEW.id
  FROM task_dependencies td
  WHERE td.depends_on = NEW.id;
END;

-- Auto-enqueue review when task review is requested
CREATE TRIGGER auto_enqueue_review
AFTER INSERT ON gate_checkpoints
WHEN NEW.gate_type = 'task-review' AND NEW.status = 'pending'
BEGIN
  INSERT INTO review_queue (task_id, requested_by)
  SELECT NEW.task_id, t.assigned_to
  FROM tasks t WHERE t.id = NEW.task_id;
END;

-- Auto-cleanup stale agent presence (>5 min)
CREATE TRIGGER cleanup_stale_presence
AFTER INSERT ON agent_presence
BEGIN
  DELETE FROM agent_presence
  WHERE heartbeat < (strftime('%s','now') - 300);
END;

-- Set review SLA (24h default)
CREATE TRIGGER set_review_sla
AFTER INSERT ON gate_checkpoints
WHEN NEW.gate_type = 'task-review' AND NEW.status = 'pending'
BEGIN
  INSERT INTO gate_sla (gate_id, deadline)
  VALUES (NEW.id, datetime('now', '+24 hours'));
END;
```

## Phase 4: Changelog Generation

**Goal:** Derive changelog from SQLite data — no manual writing.

### 4.1 CLI

```bash
acts changelog --story <id> [--format md|json|keepachangelog] [--since <date>]
```

### 4.2 Generation Rules

| Section | Source Query |
|---|---|
| **Added** | Tasks `status=DONE` with labels containing `feat` or title matching `feat:`/`add`/`implement` |
| **Fixed** | Tasks `status=DONE` with labels containing `bug`/`fix` or title matching `fix:` |
| **Changed** | Tasks `status=DONE` with labels containing `refactor`/`change` |
| **Decisions** | `decisions` table for story's tasks |
| **Rejected approaches** | `rejected_approaches` for story's tasks |
| **Reviewers** | `gate_checkpoints` where `gate_type='task-review'` and `status='approved'` |
| **Files touched** | `task_files` JOIN DONE tasks |

## Phase 6: Story Semver on Merge

**Goal:** Auto-compute and tag story versions on merge.

### 6.1 Semver Computation

Rules (conventional commits style):
- Any task with `breaking` label → major bump
- Any task with `feat` label → minor bump
- Only `fix`/`chore` labels → patch bump
- Default: patch bump from last story version

```zig
fn computeSemver(allocator: std.mem.Allocator, tasks: []Task, current: Semver) !Semver {
    var has_breaking = false;
    var has_feat = false;
    for (tasks) |t| {
        if (t.labels) |labels| {
            if (std.mem.indexOf(u8, labels, "breaking")) |_| has_breaking = true;
            if (std.mem.indexOf(u8, labels, "feat")) |_| has_feat = true;
        }
    }
    if (has_breaking) return .{ .major = current.major + 1, .minor = 0, .patch = 0 };
    if (has_feat) return .{ .major = current.major, .minor = current.minor + 1, .patch = 0 };
    return .{ .major = current.major, .minor = current.minor, .patch = current.patch + 1 };
}
```

### 6.2 Merge Flow (in `handleStoryMerge`)

1. Compute semver from task labels
2. Write `semver` + `released_at` to stories table
3. Generate changelog entry
4. Append to repo `CHANGELOG.md`
5. Create git tag `story/<id>-v<semver>`
6. All in one transaction — if any step fails, rollback

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| WAL file grows unbounded | Disk space | Auto-checkpoint in `acts validate`; manual `acts db checkpoint` |
| Cross-story trigger false positives | Blocks legitimate work | `acts override request` protocol (existing) |
| Maintenance story becomes dumping ground | Loss of traceability | `acts task move` to promote to proper story |
| Agent presence stale entries | Misleading info | Auto-cleanup trigger (5 min TTL) |
| JSON escaping breaks existing output | Breaking change | Only affects malformed data; valid data unchanged |
| Worktree path collisions | Story creation fails | Check path exists before creating; suggest alternative |
